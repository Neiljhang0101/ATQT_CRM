## 1. 變更背景與原因 (Why)

在 CR-001 雙帳號架構完成後，實際接入 BingX API 時發現三個問題：

1. **傭金 API 簽名失敗 (code=100001)**：`getAgencyCommission` 呼叫時額外附帶 `recvWindow=5000`，此參數被 BingX 傭金端點拒絕，造成伺服器端計算的 signature 與傳入值不符。
2. **子帳號資產 API 無權限 (code=100400)**：`/openApi/v3/subAccount/assets` 對代理商帳號不開放，造成帳戶餘額永遠顯示 `null`。
3. **欄位同步不完整**：API 回傳的 `inviterSid`、`directInvitation`、`inviteCode`、`ownInviteCode`、`userLevel` 等欄位，原實作未寫入 store，客戶詳細頁相關欄位全部顯示 `--`。

---

## 2. 規格變更對照 (What)

### 2.1 傭金 API 改用純 `fetch()` 函式

| 項目 | 舊實作 | 新實作 |
|---|---|---|
| 呼叫方式 | axios instance + 攔截器 | `commissionFetch()` 純函式，直接使用 `window.fetch` |
| 簽名參數 | 包含 `recvWindow=5000` | 不含 `recvWindow` |
| Key 傳遞方式 | 閉包於 axios instance | 明確傳入 `(apiKey, secretKey, params)` |
| 匯出方式 | `getDailyCommissionDetails()` | `commissionFetch()` + `OLD_API_KEY` / `NEW_API_KEY` 等常數 |

**根本原因：** axios module-level instance 在 Vite HMR 期間不會重新初始化，每次修改攔截器後須強制清除 `node_modules/.vite` 快取才能生效，維護困難。改用純函式可完全規避此問題。

### 2.2 帳戶餘額改用 `balanceVolume`

| 項目 | 舊實作 | 新實作 |
|---|---|---|
| 資料來源 | `/openApi/v3/subAccount/assets`（無權限） | 下線名單 API 回傳的 `balanceVolume` 欄位 |
| 精度 | N/A（永遠 null） | `parseFloat(item.balanceVolume)` |

### 2.3 全欄位 API 對應

新增以下欄位的自動同步（`fetchAccount` 與 `fetchInviteesOnly` 均已更新）：

| API 欄位 | store 欄位 | 說明 |
|---------|-----------|------|
| `inviterSid` | `inviter_uid_code` | 值為 `0` 時視為無邀請人，存 `null` |
| `directInvitation` | `invite_type` | `true` → `'直接'`，`false` → `'間接'` |
| `inviteCode` | `bound_invite_code` | 空字串視為 `null` |
| `ownInviteCode` | `own_promo_code` | 空字串視為 `null` |
| `userLevel` | `bingx_vip_level` | 值為 `0` 時視為無等級，存 `null` |
| `balanceVolume` | `total_assets` + `balance` | 字串轉 float |
| `registerDateTime` | `bingx_register_date` | ms 時戳 → `YYYY-MM-DD` |
| `sourceLabel` | `account_type` | `'old'` / `'new'` |

### 2.4 同步按鈕整併為下拉選單

| 項目 | 舊實作 | 新實作 |
|---|---|---|
| UI | 4 個獨立按鈕（下線名單、傭金明細、帳戶餘額、全部同步）| 1 個「同步資料 ▾」下拉選單 |
| 狀態管理 | 各別 `ref` | 同 + `anySyncing` computed 統一控制 disabled |
| 指令路由 | 各 `@click` 直接呼叫 | `handleSyncCommand(cmd)` 統一路由 |

## 4. 已知限制 (Known Constraints)

1. **首次入金時間**：BingX API 僅回傳 `deposit: bool`，無法取得具體時間戳。`first_deposit_time` 仍需透過 USER-INFO XLSX 匯入。
2. **`commissionFetch` 僅透過 Vite Proxy**：若未來部署至雲端，需在後端中介層另行實作相同簽名邏輯。
3. **`balanceVolume` 精度**：代理商 API 的 `balanceVolume` 為帳號總資產估算值，與實際 USDT 現貨餘額可能有微小誤差。
